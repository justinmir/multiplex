# M1.4 — IpcDataSource replaces MockDataSource; add error state

## Goal
Create `IpcDataSource` that fetches data from disk via IPC (replacing MockDataSource). Add error state to DataProvider so callers can surface load failures. The app shows the same seeded content, but sourced from `db.json` on disk.

## Files changed

### `packages/renderer/src/lib/data/IpcDataSource.ts` (new file)
- Implements `DataSource` by delegating to IPC channels via `call()`:
  - `listProjects()` → `"projects:list"` (void request)
  - `getProject(id)` → `"projects:get"` with `{ id }`
  - `listStandaloneSessions()` → `"sessions:list"` with `{ projectId: null }` (filters to standalone only)
  - `getSession(id)` → `"sessions:get"` with `{ id }`

### `packages/renderer/src/lib/data/DataProvider.tsx`
- Removed `MockDataSource` import and lazy-default pattern (`useState(() => new MockDataSource())`)
- Added `error: string | null` to `DataContextValue` interface and context state
- `loadData()` now catches errors from IPC calls and stores the message in `error` state
- Exported `useDataLoading()` hook returning `{ loading, error }` for top-level UI consumption

### `packages/renderer/src/main.tsx`
- Imports `IpcDataSource` and passes a new instance as `source={new IpcDataSource()}` to `<DataProvider>`
- No more implicit MockDataSource fallback — the app is now fully IPC-backed

## Design decisions
- `MockDataSource` retained on disk (not deleted) for potential future use in testing
- `listStandaloneSessions()` calls `"sessions:list"` with `{ projectId: null }` per the M1.3 handler contract, which returns only sessions not attached to any project
- Error state is captured but not rendered inline — AppShell can consume `useDataLoading()` when a proper error/loading UI is added

## Validation

npm run validate    # typecheck + lint (0 errors) + build: OK
npm run screenshot  # app boots, renders seeded data from db.json via IPC: OK
