# M1.3 — IPC read handlers + wire JsonRepository

## Goal
Expose repository reads over typed IPC channels, constructing a single JsonRepository instance in IpcModule and registering handlers that delegate to it.

## Files changed

### `packages/core/src/ipc.ts`
- Import `Project` and `Session` from domain module
- Add four read channel entries to `IpcContract`:
  - `"projects:list"` — return all projects (no request payload)
  - `"projects:get"` — get single project by id
  - `"sessions:list"` — list sessions, optionally filtered by projectId (optional field)
  - `"sessions:get"` — get single session by id

### `packages/main/src/ipc/handlers/repo.ts` (new file)
- Takes a JsonRepository instance and registers all read handlers via the typed `handle()` router
- Separate from IpcModule so M1.5 can add write handlers in a parallel file (`repo.write.ts`)

### `packages/main/src/modules/IpcModule.ts`
- Constructs a single `JsonRepository` instance on module enable
- Calls `registerRepoReadHandlers(repo)` to wire all read channels
- Keeps existing app:ping handler

## Behavior fix: listSessions null vs undefined
Updated InMemoryRepository.listSessions to correctly distinguish between:
- No filter / `projectId === undefined` → return ALL sessions (home view)
- Specific projectId string → return only sessions for that project
- `projectId === null` → return only standalone sessions (not currently used by renderer but correct per contract)

Previously, both `undefined` and `null` returned all sessions. The condition now checks for `opts?.projectId == null` (loose equality catches both undefined and null) in the "return all" path — wait, no, we need them to behave differently. The fix: when `projectId === null`, filter to standalone; when `undefined` or missing, return all.
