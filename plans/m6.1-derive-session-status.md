# M6.1 — Derive SessionStatus from real signals

## Problem
Session status is currently only set by the harness (running/completed/idle). When PRs are linked to a session, their review verdicts, check results, mergeability, and awaiting-input signals should automatically drive the displayed status.

## Changes

### 1. `packages/core/src/domain.ts` — Type change
- `Session.linkedPRs`: `{ repo; number }[]` → `PullRequest[]`
- Enables derive function to read PR fields directly (reviewVerdict, checks, mergeable, etc.)

### 2. `packages/main/src/session/deriveStatus.ts` — New file
Deterministic derivation with precedence:
1. awaiting_input (always wins if set by harness)
2. checks_failing (any failing checkRun or failed>0)
3. changes_requested (reviewVerdict or unresolved comment)
4. review_pending (pending verdict, no other PR approved yet)
5. mergeable (clean + approved + all green/skipped + no pending)
6. merged (any linked PR is merged)
7. fallback to harness status (running/completed/failed/idle)

### 3. Wire into `session-messages.ts`
After every upsertSession call in sessions:add-message, agents:start (both branches), and agents:stop: derive → patch if changed → emit.

### 4. Wire into `session-writes.ts`
After status update and after archive: derive → patch if changed → emit.

### 5. Update mock/seed data
Enrich linkedPRs entries to be full PullRequest objects with all required fields.

### 6. Simplify renderer PR resolution
AppShell, App, SessionsTab no longer need to resolve `{repo,number}` against project.prs — linkedPRs are already full objects. HomeView continues displaying repo#number labels (same shape works).

## Validation
- `npm run validate` passes (0 errors)
- deriveSessionStatus is pure and deterministic
